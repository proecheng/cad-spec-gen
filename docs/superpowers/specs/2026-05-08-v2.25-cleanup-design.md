### I-3：`evidence_token` 取原文切片（双改）

#### 改动位置

`tools/product_goal_parser.py:_extract_single_kpi` **两处都要改**（reviewer 抓到只改一处会导致下游 audit 仍部分失真）：

1. `KpiExtraction.evidence_token=match.group(0)`（line 268，函数返回值）
2. `evidence.append({..., "token": match.group(0), ...})`（line 259，evidence list `"token"` 字段）

#### 实现（修正 fallback 设计）

reviewer B 抓到原 fallback 设计有 bug：finditer 跑在 `normalized` 上（line 234 `compiled.finditer(normalized)`），`match.start()/end()` 是 normalized 索引域；直接用此索引去切 `raw_text` 在 NFKC 拆分场景（如 `⅔ → 2/3`）会错位。原 spec `len(raw_token) != end-start` 永远 false，fallback 永不触发。

修正：在归一化时检查 raw 与 normalized 长度，相等才走原文切片：

```python
def _raw_token_for_match(
    raw_text: str,
    normalized: str,
    match: re.Match[str],
) -> str:
    """从原文取 match 区间对应的子串。

    NFKC 在 ASCII 路径与全角→半角场景下保持 char 数 1:1（len(raw)==len(normalized)），
    此时 match.start()/end() 在两个串上索引等价，可直接切 raw_text。
    极少数 NFKC 拆分（⅔ → 2/3、㎡ → m2）会让 normalized 比 raw_text 长，索引域错位
    → fallback 到 match.group(0)（normalized 串上 token，与现状等价不优于 normalized）。
    """
    if len(raw_text) == len(normalized):
        return raw_text[match.start():match.end()]
    return match.group(0)
```

调用点（双改）：
```python
# evidence list "token" 字段（line 259）
raw_token = _raw_token_for_match(raw_text, normalized, match)
evidence.append({
    "token": raw_token,
    "matched": kpi_name,
    ...
})
# KpiExtraction.evidence_token（line 268）
return KpiExtraction(
    ...,
    evidence_token=raw_token,
    ...
)
```

#### 用户场景

输入 `"载荷 ５０ｋｇ 平台"`（全角数字 + 全角字母）：
- NFKC 后：`"载荷 50kg 平台"`（len 相等：全角 char 1:1 → 半角 char）
- 旧实现：`evidence_token = "50kg"`（用户没看见自己打的 ５０ｋｇ）
- 新实现：`evidence_token = "５０ｋｇ"`（保留原文，audit 友好）

输入含 NFKC 拆分（罕见，如 `"载荷⅔kg"`）：
- NFKC 后：`"载荷2/3kg"`（len 不等）
- fallback：`evidence_token = match.group(0)`（normalized 串上 token，等价旧行为）

```python
_CJK_PATTERN = re.compile(r"[一-鿿㐀-䶿]")  # CJK Unified + Ext-A
```

#### 新增 `_classify_unsafe_reason`（优先级修订）

reviewer C 抓到原优先级反例：用户在中文里随手敲反斜杠（如 `"做\升降"` 或 `"高\"精度\""` 转义引号）会被误分到 `windows_path`。修订：

1. **CJK 优先于 windows_path**：含中文一律走 chinese_text 提示
2. **windows_path 收紧**：只看 `^[A-Za-z]:[\\/]` 起始（盘符冒号），不再 OR `"\\" in value`

```python
def _classify_unsafe_reason(value: str) -> str:
    """分类 user_text 不安全原因，给出精确下一步提示。

    返回 'safe' | 'windows_path' | 'chinese_text' | 'special_chars'。

    优先级（互斥）：safe > chinese_text > windows_path > special_chars
    """
    if not value:
        return "safe"
    if _SAFE_CLI_PATTERN.fullmatch(value):
        return "safe"
    if _CJK_PATTERN.search(value):
        return "chinese_text"
    if _WIN_PATH_PATTERN.match(value):
        return "windows_path"
    return "special_chars"
```

`"D:\Work\做平台.md"`（中文 + 盘符路径）会归 `chinese_text`——用户主要痛点是中文（forward slash 改写后仍有中文段），文案"用 --confirm-X flag 直接传值"对此场景更合适。

```python
) -> dict[str, Any]:
    """检查 next_action.preview_cli 是否含 user text；含且 unsafe 则降级。

    不变量（M-4 修订）：preview_cli_unsafe 与 unsafe_reason 同生共灭——
    不存在 preview_cli_unsafe=True 但 unsafe_reason 缺失的状态；
    safe 路径下两个字段都不写。
    """
    raw_text = getattr(parse_result, "raw_text", None) or ""
    if not raw_text:
        return action
    cli = action.get("preview_cli")
    if not isinstance(cli, str):
        return action
    if raw_text not in cli:
        return action
    reason = _classify_unsafe_reason(raw_text)
    if reason == "safe":
        return action
    action["preview_cli_unsafe"] = True
    action["unsafe_reason"] = reason
    action["preview_cli"] = _UNSAFE_MESSAGE_TEMPLATES[reason]
    return action
```

#### 不变量声明（M-4 + reader 兼容性）

- `unsafe_reason` 字段值域闭集：`{"windows_path", "chinese_text", "special_chars"}`
- `preview_cli_unsafe=True` ⇔ `unsafe_reason` 存在；safe 路径下两个字段都不写
- 既有 `tests/test_project_goal_guide.py:126-141`（仅断言 `preview_cli_unsafe is True`）保持兼容
- 新增字段 `unsafe_reason` 不破坏既有 reader（grep 全仓未发现 PROJECT_GUIDE.json schema validator；新字段是 additive 无禁用 enum）
