**日期**: 2026-04-12

**主题**: 消除 `src/cad_spec_gen/data/` 双副本 + 移除 v2.9.0 遗留的 P2 regex fallback

---

## 背景

v2.9.2 用 `tests/test_data_dir_sync.py` 止住了 data/ mirror drift，但根因未解——`hatch_build.py` 在 wheel build 时把 root 源文件复制到 `src/cad_spec_gen/data/`，两份拷贝都 git-tracked，每次 build 后 git status 冒假 `M`。

同时，v2.9.0 引入的 section-header walker 已经过 v2.9.0-2.9.2 三个版本的真实文档验证，`_legacy_p2_regex_block` 和 `CAD_SPEC_WALKER_ENABLED` feature flag 可以安全移除。

### `scripts/dev_sync.py`

**职责**: 将 root 源文件同步到 `data/`，与 `hatch_build.py::CustomBuildHook.initialize()` 产出一致。

**设计要点**:
1. **常量来源**——用 `ast.parse()` 从 `hatch_build.py` 提取 `_PIPELINE_TOOLS`、`COPY_DIRS`、`TOP_LEVEL_FILES` 等常量（与 `test_data_dir_sync.py` 相同策略）。不直接 import `hatch_build`，因为开发环境不一定安装了 `hatchling`（它是 `[build-system] requires` 依赖，非运行时依赖）。`SHARED_TOOL_FILES` 从同目录的 `cad_paths.py` 直接 import（它无 hatchling 依赖）。
2. **幂等执行**——每个文件先 MD5 比较，内容一致则跳过；只有实际写入的文件才计入"有变更"。跑多次结果一样，不会因文件已存在就误报变更。
3. **可被 pre-commit 和 test 调用**——提供 `sync(root: Path) -> list[Path]` 函数（返回实际更新的文件列表）+ `if __name__ == "__main__"` CLI 入口。
4. **覆盖 hatch_build 的全部同步项**：python_tools、codegen/config/templates 目录、commands/zh、knowledge zh 文件、system_prompt.md、TOP_LEVEL_FILES。
5. **退出码**：有文件被更新 → exit 1（pre-commit 约定：hook 修改了文件）；无变更 → exit 0。幂等判定基于 MD5，不依赖文件 mtime。

**不做的事**:
- 不做 diff 报告
- 不做 git add
- 不做反向同步（data/ → root）

### `.pre-commit-config.yaml`

```yaml
repos:
  - repo: local
    hooks:
      - id: dev-sync
        name: Sync root sources to data/
        entry: python scripts/dev_sync.py
        language: python
        always_run: true
        pass_filenames: false
```

- `always_run: true`：不依赖文件变更检测，每次 commit 都确保同步
- `language: python`：用当前环境 Python，不创建额外 venv
- 仅此一个 hook，不引入 ruff 等其他 hook（scope 不膨胀）

### `tests/test_data_dir_sync.py` 重写

从 "git drift 检测" 变为 "dev_sync.py 正确性验证"：

1. 参数化逻辑不变——继续用 `ast.parse()` 从 `hatch_build.py` 派生同步对
2. session-scope fixture 跑一次 `sync()`，47+ 个 case 共享结果
3. 每个 case 断言 data/ 文件存在且 MD5 与 root 一致
4. 保留 `test_sync_pair_count_is_nontrivial`（ast 健康检查）
5. 保留 `test_python_tools_has_no_stray_files`（反向哨兵）

## 工作块 2：legacy P2 清除

### `cad_spec_extractors.py` 变更

**删除**:
- `_WALKER_ENABLED` 变量定义（L14-16）
- `elif not _WALKER_ENABLED` 分支 + `else` 分支（L1217-1229）
- `_legacy_p2_regex_block()` 函数（L1288-1336）

**简化后的 P2 分支**:

```python
# 之前（3 个分支）：
if _WALKER_ENABLED and _WALKER_AVAILABLE and bom_data:
    # walker
elif not _WALKER_ENABLED:
    # flag 关闭 → legacy regex
else:
    # walker import 失败 → legacy regex fallback

# 之后（2 个分支）：
if _WALKER_AVAILABLE and bom_data:
    # walker（不变）
else:
    # walker 不可用 → 跳过 P2，log.warning
```

**行为变化**:
- `CAD_SPEC_WALKER_ENABLED=0` 环境变量不再被读取
- walker import 失败时 P2 直接跳过，不 fallback 到 regex
- 安全性：v2.9.0-2.9.2 已验证 walker 优于 regex

**测试影响**: 无——现有 walker 测试覆盖主路径，无测试依赖 `CAD_SPEC_WALKER_ENABLED=0`。

## CI 影响

修改 `.github/workflows/tests.yml`，在两个 job（`test` 和 `regression`）的 pytest 步骤前各加一步：

```yaml
- name: Sync data/ mirrors
  run: python scripts/dev_sync.py
```

当前 CI 不做 `pip install -e .`（直接 `pip install pytest ...` + 跑测试），`git rm --cached` 后 checkout 出来的仓库不含 mirror 文件。`dev_sync.py` 填充它们，确保测试能找到依赖的 data/ 内容。CI 不装 pre-commit。

## 升级迁移（v2.9.x → v2.10.0）

现有 clone 的开发者 `git pull` 到 v2.10.0 后，data/ mirror 文件仍在磁盘上且被 git tracked（`.gitignore` 只影响未 tracked 的文件）。需要一次性清理：

```bash
git pull origin main
# 移除已被 .gitignore 但仍在 index 中的 mirror 文件
git rm --cached -r src/cad_spec_gen/data/python_tools/
git rm --cached -r src/cad_spec_gen/data/codegen/
git rm --cached -r src/cad_spec_gen/data/templates/
git rm --cached -r src/cad_spec_gen/data/config/
git rm --cached -r src/cad_spec_gen/data/commands/zh/
git rm --cached src/cad_spec_gen/data/knowledge/skill_cad_help_zh.md
git rm --cached src/cad_spec_gen/data/knowledge/skill_mech_design_zh.md
git rm --cached src/cad_spec_gen/data/system_prompt.md
git rm --cached src/cad_spec_gen/data/parts_library.default.yaml
```

实际上，v2.10.0 的发布 commit 本身就包含了 `git rm --cached`，所以其他开发者 `git pull` 后 index 自动同步——不需要手动操作。磁盘上残留的文件会被 `.gitignore` 忽略，不影响 `git status`。`RELEASE_v2.10.0.md` 中仍需说明此行为变化。
