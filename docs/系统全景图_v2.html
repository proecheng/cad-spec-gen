概览

从 CAD 设计文档到照片级渲染图，端到端 AI 增强 + LLM 评分自动验收闭环。

~99% 完工 · 15 PR 连续一次过 CI

本项目把工程师的 设计文档（中文 Markdown）转换为可交付的 照片级 3D 渲染图，跨 6 阶段管线（spec → codegen → build → render → enhance → annotate/deliver），AI 评分自动验收闭环把人工 review 成本降至最低。Windows-only，CAD 后端走 SolidWorks/CadQuery，渲染走 Blender，AI 增强走多 backend（gemini/fal/comfyui/engineering），评分走 OpenAI 兼容 vision LLM。

不变约束

范围限定

傻瓜式用户视角

默认用户外行（工程师不是 AI 专家）；只问业务语义（"做一个升降平台"），不问技术选型；wizard / autopilot 把多步骤合一个命令。
UX 北极星

AI 不能补缺件

AI 增强是"提质量"不是"造内容"。CAD 自制件几何必须由 CadQuery 真实生成；jury matches_spec 维度验证设计特征是否真在图里可见，禁止伪造照片级最终交付。

审查方法论

spec → plan → execute → review → merge 全程纪律。连续 15 PR 一次过 CI 8/8 的工程实证。

superpowers + 6 层 review

5 层 + 1 scout 审查（spec ≥ 100 行默认走）

每层抓不同 bug 类别。layer 1-4 静态分析；layer 5 是 5 角色并行 adversarial；layer 6 是 writing-plans 入口 scout grep 实际代码验证 spec 假设。

1
Self-review
写完 spec 4 项 inline check：placeholder / 内部一致 / scope / ambiguity
2
Cynical re-read
改完是否变差？隐性副作用 / 副效应
3
Code-spec 对照
spec 引用真假事实 grep + Read 验证
4
Edge-case hunter
method-driven 路径枚举边界条件，只报 unhandled paths
5
5 角色 + dry-run
Murphy 悲观 / 1 年后维护者 / SRE / QA / holistic dry-run
6
writing-plans scout
进 plan 前 grep 真实代码 hardcode 验证 spec 假设

subagent-driven 执行流

主 agent 每 plan task 派发 fresh subagent；2 阶段 review（spec compliance + code quality）；fresh context 防污染。15 PR 连续一次过 CI 实证。

阶段角色关键产物纪律

writing-plans主 agentplan 含 4-7 task / 每 step 2-5 min无 placeholder / type consistency / spec coverage
execute Task Nfresh subagentcode + test commitTDD RED→GREEN→REFACTOR / 显式 R 步

TDD 三步铁律

RED · 先写失败测试

禁止跳步直接写实现代码；先 assertion 锁定行为，pytest -v 验证 FAILED。

GREEN · 最小实现

测试通过后不得多写额外逻辑；YAGNI 严格执行；spec 没要求的不做。

REFACTOR · 显式 R 步

scope 小可标"REFACTOR: 无冗余可清，跳过"；commit message 必含 R 步声明，防"测试通过即停手"。

v2.37.5 glossary

10 项核心术语 glossary（CLAUDE.md 同步）

1 · 北极星 5 gate

零配置 / 稳定可靠 / 结果准确 / SW 装即用 / 傻瓜式操作。任何新 plan 必过。

2 · v2.25+ tag-only release

纯 git tag + GitHub Release notes 发布；不 bump pyproject.toml（停留 2.24.0）。用户 pip install git+https://...@vX.Y.Z。

3 · canonical / mirror

tools/jury/*.py = canonical（git tracked）；src/cad_spec_gen/data/tools/jury/*.py = mirror（gitignored，scripts/dev_sync.py 同步）。hatch_build COPY_DIRS 打包。

4 · §11 vs §12 follow-up 轨道

§11 = 项目级 STATUS doc follow-up；§12 = 单 spec PR-self follow-up。两个独立轨道不混。

5 · pure refactor / no-RED phase

refactor 或 docs-only PR 用既有测试作 GREEN safety net；TDD R-only 模式；commit body 必显式标注。

6 · 5 层 + 1 scout 审查

spec ≥ 100 行默认跑：self / cynical / code-spec / edge-case / 5 角色 + dry-run 五层 + writing-plans 入口 scout grep。

7 · subagent-driven 模式

主 agent 每 task 派发 fresh subagent + 2 阶段 review（spec compliance + code quality）；连续 15 PR 一次过 CI 实证。

8 · layer 6 grep AC predicates

spec AC 用 exclusion-zone（grep -v）或 indent-anchor；OR pattern 用 grep -cE "X|Y" 显式 ERE 跨平台可靠。

9 · sw-smoke CI flake

sw-smoke.yml 的 actions/upload-artifact@v7 步是已知 transient flake；与 tests.yml 8 job release gate 无关。

10 · plan-drift 5 分类

spec 写时假设漂移：(a) API 不存在 (b) 路径假设错 (c) 测试 helper 误用 (d) 实现细节 bug (e) 参数签名。plan 第 0 task scout grep 防御。
